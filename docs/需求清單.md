## 2. 登入（正式版待實作）

> **線上現況**：見 **§0** — 已用單一共用帳號 + RLS 過渡，非本章完整規格。

### 2.1 安全模型（已決定）

**方案 B：Supabase Auth + RLS**

- 資料庫層強制隔離，業務帳號無法用 anon key 讀寫他人資料。
- 需啟用 RLS、撰寫 policy、前端改為登入後才存取 Supabase。
- 建立帳號需後端（API / Edge Function）使用 service role，不可暴露在前端。

### 2.2 預設未登入（已決定）

**可試算但不存**

| 允許 | 禁止 |
|------|------|
| 進入帳本 UI | 讀寫 Supabase |
| 貼報價單 URL、呼叫 `/api/fetch-quote` | 資料落地、再同步寫入 |
| 當次試算結果（記憶體） | 季度倍率寫入 |
| **匯出當次試算 CSV** | |

- 關分頁或重整 → 試算資料消失。
- UI 顯示提示：「試算模式，資料不會儲存」。

### 2.3 登入後

- 載入 Supabase 資料，恢復現有讀寫流程。
- 依帳號角色與業務別名過濾案件（見 §4）。
- RLS 與前端篩選雙重保障。

### 2.4 起始帳號

| 登入帳號 | 密碼 | 角色 | 顯示名稱 | 說明 |
|----------|------|------|----------|------|
| `ivy` | `123456` | 超級管理員 | （你提供） | seed 時 `must_change_password=true`；首次登入須改密 |

> **Auth email 規則（見 §11 衝突 #1）**  
> - 登入畫面只打**登入帳號** `ivy`，不打 email。  
> - ivy 的 Supabase Auth email 使用你提供的**真實 email**（實作前提供）。  
> - 一般業務帳號 Auth email 格式：`{login_username}@{AUTH_EMAIL_DOMAIN}`（**網域實作前再定**）。

### 2.5 登入狀態保存（已決定）

- 提供 **「記住我」** 勾選。
- 未勾選：`sessionStorage`（關瀏覽器需重登）。
- 已勾選：`localStorage`（關瀏覽器仍保持登入）。

### 2.6 登入畫面（已決定）

- 使用者輸入 **登入帳號 + 密碼**（不直接填 email）。
- **登入帳號**與**顯示名稱**分開（見 §5.2）：
  - 登入帳號：英文/數字，全系統唯一（如 `ivy`、`amy01`）。
  - 顯示名稱：可中文（如 `Ivy 蔣慶瑤`），用於側欄、問候等 UI。
- 後端查 `user_profiles.login_username` → 取得對應 Supabase Auth email 再 signIn（見 **§2.9** `POST /api/auth/login`）。

### 2.7 未登入可逛的頁面（已決定）

- **全部頁面可進**（帳本、季度倍率、獎金規則），唯讀試算模式。
- 不寫入 Supabase；倍率頁不可編輯；無管理頁入口。

### 2.8 `/api/fetch-quote` rate limit（已決定）

訪客仍可試算，但需基本防濫用。實作於 [`api/index.py`](../api/index.py)（in-memory sliding window；Vercel 多 instance 為 best-effort）。

| 情境 | 規則 |
|------|------|
| **未登入** | 同一 IP **每分鐘最多 20 次**；超限回 `429` + 友善訊息 |
| **未登入** | **不限制** |
| 網域白名單 | 維持僅 `quote.saiens.tw` |
| 已登入免限流 | 請求可帶 `Authorization: Bearer <JWT>`，後端驗證後跳過限流 |

### 2.9 後端 API（service role，已決定）

`user_profiles` **不可**對 anon 開放 SELECT。登入與建帳皆走後端（[`api/index.py`](../api/index.py) 擴充），使用 `SUPABASE_SERVICE_ROLE_KEY`（僅 Vercel env，不進前端）。

| 端點 | 用途 | 誰可呼叫 |
|------|------|----------|
| `POST /api/auth/login` | 查 `login_username` → 驗證 active → Supabase signIn；錯誤統一「帳號或密碼錯誤」 | 公開 |
| `POST /api/auth/change-password` | 改 Auth 密碼 + service role 設 `must_change_password=false` | 已登入 |
| `GET /api/admin/users` 等 | 列表 + 建立 Auth user + `user_profiles`、綁定業務 | 僅 ivy |
| `POST/PATCH/DELETE` admin | 業務、別名、帳號 CRUD | 僅 ivy |
| 停用 / 重設需改密 | 更新 `active`、`must_change_password` | 僅 ivy |

**登入流程（固定）**：前端 → `/api/auth/login` → 後端 GoTrue password grant → 回 token + profile → 前端 `setSession` → 若需改密則 modal，否則 `ensureLoaded()`。

前端登入流程：送登入帳號 + 密碼至 `/api/auth/login` → 取得 session → 依 `must_change_password` 決定是否先走改密 modal。

## 4. 帳號 ↔ 業務（待實作）

### 4.1 對應關係（已決定）

- **一對一**：一個登入帳號對應一個業務主檔（`user_profiles.sales_rep_id`）。
- **ivy**：角色為超管，**同時綁定自己的業務主檔**（供預設篩選）；權限上仍可看全部業務與「全部」選項。

### 4.2 案件歸屬判斷（已決定）

**ivy 在管理頁維護「業務別名」對照表**

1. 每位業務有「顯示名稱」+ 多個「別名」。
2. 報價單寫入 `bonus_records.sales_rep` 為原始字串（如 `Ivy 蔣慶瑤`）。
3. 歸屬規則：`sales_rep` 與該業務任一別名 **字串完全相等** → 算該業務的案件。
4. 新別名僅 ivy 可維護；業務帳號不可改對照表。
5. 若報價單出現新別名、尚未建檔 → **僅 ivy 看得到**（「未歸屬」篩選）；案件列顯示 **「未歸屬」** badge，ivy 點選後指定業務 → 自動將該 `sales_rep` 字串加入該業務別名表。
6. **未歸屬定義**：`trim(sales_rep) ≠ ''` 且別名表無匹配；**空字串不算未歸屬**。
7. **未歸屬案件不計入**脈絡列 KPI（應計/實領）與統計區塊；僅在表格列出供 ivy 處理。
8. **rep 不可手改** `sales_rep` 欄（唯讀）；ivy 可改；再同步仍更新業務。
9. **空 `sales_rep`**：`trim(sales_rep)=''` **不算未歸屬**；**僅 ivy（super）可寫入 DB**；rep 儲存前若為空則阻擋並提示：「此報價單無案件業務，無法加入你的帳本；請聯絡管理員或請 ivy 處理。」

### 4.3 季度倍率權限（已決定）

| 操作 | 未登入 | 一般業務 | ivy |
|------|--------|----------|-----|
| 讀取倍率、參與計算 | 記憶體預設值試算 | ✅ | ✅ |
| 編輯倍率 | ❌ | ❌ | ✅ |

倍率為全站共用一份，非每人一份。

**rep 新增案件時**：若該回簽季度在 DB 尚無倍率列，**不寫入** `quarter_multipliers`；計算 fallback 記憶體預設 `(1,1,1,1)`。僅 ivy 在倍率頁持久化倍率。

### 4.4 首次改密碼（已決定）

- 凡 `user_profiles.must_change_password = true`（含 seed 的 **ivy**、新業務帳號、或 ivy 重設為需改密狀態）→ 登入成功後跳出 **modal** 強制改密碼。
- 改完前**不能進入主系統**（不可關閉 modal 略過）。
- 改密成功後設 `must_change_password = false`；已改過密碼的帳號正常進入。

### 4.5 刪除案件（已決定）

- **一般業務**：只能刪除自己別名對到的案件。
- **ivy**：可刪除任何案件。
- **未登入**：無刪除（試算列僅記憶體，關頁即消失）。

### 5.2 帳號欄位（已決定）

建立帳號時 ivy 填寫：

| 欄位 | 規則 | 範例 |
|------|------|------|
| **登入帳號** | 英文/數字，唯一，不可空白 | `ivy`、`amy01` |

### MVP 功能

1. **業務主檔**：新增 / 停用業務（顯示名稱）。
2. **別名管理**：為每位業務維護別名列表（對應報價單「案件業務」欄）。
3. **帳號管理**：建立登入帳號、角色（`super` / `rep`）、綁定業務（rep 必填）；ivy 設定**初始密碼**，業務**首次登入必須修改密碼**。
4. **停用帳號**：停用後無法登入。
5. **重設需改密**：ivy 可將帳號設回 `must_change_password = true`。

## 7. 資料模型（草案）

```sql
-- 業務主檔
sales_reps (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  display_name  text NOT NULL,
  active        boolean NOT NULL DEFAULT true,
  created_at    timestamptz NOT NULL DEFAULT now()
)

-- 別名（ivy 維護；全域唯一）
sales_rep_aliases (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  sales_rep_id  uuid NOT NULL REFERENCES sales_reps(id),
  alias         text NOT NULL UNIQUE,
  created_at    timestamptz NOT NULL DEFAULT now()
)

-- 使用者設定檔（連 auth.users）
user_profiles (
  user_id         uuid PRIMARY KEY REFERENCES auth.users(id),
  login_username  text NOT NULL UNIQUE,
  display_name    text NOT NULL,
  auth_email      text NOT NULL UNIQUE,   -- Supabase Auth 用；ivy=真實 email，其他={login}@{domain}
  role            text NOT NULL,
  sales_rep_id    uuid NOT NULL REFERENCES sales_reps(id),
  must_change_password boolean NOT NULL DEFAULT false,
  active          boolean NOT NULL DEFAULT true
)
```

- `bonus_records.sales_rep`：維持 text，存報價單原始字串（**不**加 `sales_rep_id` FK，見 §12）。
- `quarter_multipliers`：全域共用；RLS 依角色區分讀寫。
- seed ivy：`must_change_password = true`、綁定業務主檔 + 初始別名。

### 7.1 RLS 與 migration（已決定）

- 新增 [`supabase/migrations/`](../supabase/)（或 `rls.sql`）：`enable row level security` + policies。
- 更新 [`supabase/schema.sql`](../supabase/schema.sql)：移除「Personal use: disable RLS」；新環境跑完整 migration。

| 表 | anon | rep（authenticated） | super |
|----|------|----------------------|-------|
| `bonus_records` | 無權 | CRUD 別名匹配列；**`trim(sales_rep)=''` 僅 super 可寫** | 全部 |
| `quarter_multipliers` | 無權 | 讀 | 讀寫 |
| `sales_reps` / `aliases` | 無權 | 讀（計算歸屬用） | 讀寫 |
| `user_profiles` | 無權 | 讀自己 | 讀寫全部 |

歸屬判斷在 RLS 可透過 `sales_rep_aliases` join 或 security definer helper function 實作。

**上線 RLS 前**：既有案件應已清空，或 ivy 先建好別名；否則 rep 可能看不到舊資料。

## 11. 需求衝突釐清

以下為對話中曾出現矛盾或易混淆之處，**以本節為準**。

| # | 衝突 | 釐清後結論 |
|---|------|------------|
| 1 | ivy 用「真實 email」 vs 全员 `{login}@internal` | **ivy**：Auth email = 你提供的真實 email；登入仍打 `ivy`。 **一般帳號**：Auth email = `{login_username}@{AUTH_EMAIL_DOMAIN}`（網域待定）。 |
| 2 | ivy「超管不綁業務」 vs「預設看自己」 | ivy **綁自己的** `sales_reps` 一筆；`role=super` 決定能否看全部/進管理頁。 |
| 3 | ivy 預設「自己」 vs 未歸屬案件可見性 | 未歸屬列**只在** ivy 選「**未歸屬**」篩選時出現；選「自己/全部」時不混入。 |
| 4 | 未歸屬「管理頁處理」 vs「表格 badge」 | 掛別名在**帳本表格**操作；管理頁只管業務主檔/別名/帳號，不做待辦清單。 |
| 5 | `sales_reps.display_name` vs `user_profiles.display_name` | **兩個獨立欄位**；建立帳號時「綁定業務」選既有業務主檔；名稱可相同，不強制同步。 |
| 6 | 權限矩陣「ivy 看全部」 vs 預設篩選 | **RLS/權限**：ivy 可讀全部。**UI 預設**：篩選為「自己」；切「全部」才看總盤。 |
| 7 | §1「手動指定業務」 vs 已實作 | **作廢**；以自動抓報價單 + 表格可改為準。 |
| 8 | §4.4「ivy 超管不受改密限制」 vs 業界對齊 | **作廢**；**ivy 首次亦須改密**；僅 `must_change_password=false` 後正常進入。 |
| 9 | §9.4「空 sales_rep 可當未歸屬」 vs 未歸屬定義 | **作廢**；空字串 **不算**未歸屬（§4.2 #6）。 |
| 10 | §1「表格可手改 sales_rep」 vs 多帳號 | **rep 不可手改**；**ivy 可改**；再同步仍更新（§4.2 #8）。 |
| 11 | 空 sales_rep 是否可寫入 | **rep 不可**；僅 super；前端阻擋 + 提示（§4.2 #9）。 |

## 12. 不必追求（Out of scope）

以下**明確不做**；實作與驗收皆不納入：

| 項目 | 說明 |
|------|------|
| MFA / 2FA | 內部小團隊，成本效益低 |
| 稽核日誌 / 操作軌跡 | 無合規要求 |
| 密碼重設信 / Forgot password | ivy 在管理頁重設初始密碼或設需改密即可 |
| SSO / OAuth 企業登入 | 帳號數少，Supabase email+password 足夠 |
| BFF 全包 API | 維持前端 + anon key + JWT + RLS（Supabase 常規） |
| `sales_rep_id` FK 重構 | 維持 `bonus_records.sales_rep` 原始字串 + 別名表（§4.2） |
| 企業合規（SOC2、資料保留政策等） | 超出產品範圍 |
| 角色細分 | 僅 `super` / `rep` 兩種 |
